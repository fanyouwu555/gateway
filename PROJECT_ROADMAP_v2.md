## 三、核心服务深度分析与改进点

### 1. 智能路由 (SmartRouter)

**当前状态**: ✅ 基础框架良好，支持 4 种策略  
**改进方向**:
- ✅ 支持基于请求特征的动态权重调整
- ✅ Provider 级故障转移链配置
- ✅ 历史延迟数据 EMA 指数平滑（替代简单平均）

### 2. 缓存服务 (Cache)

**当前状态**: ⚠️ 内存缓存工作正常，但 Redis 读取逻辑未实现（get() 直接返回 null）  
**关键修复**:
- 🔴 将同步 `get()` 改为异步 `getAsync()`，调用方 `chat.ts` 同步修改
- 实现 `Cache-Aside` 模式：先查内存 → 再查 Redis → 最后查 Provider
- 语义缓存采用 LSH 局部敏感哈希（第一阶段先用 Token Jaccard 相似度）

### 3. 故障转移 (Failover)

**当前状态**: ⚠️ 仅支持单 API Key 健康检测，不支持跨 Provider 降级  
**关键增强**:
- 新增 `FailoverChain` 配置：`primary → fallback1 → fallback2`
- 基于错误率 + 延迟双阈值触发降级
- 自动健康恢复探测

### 4. WebSocket / 流式响应

**当前状态**: ⚠️ 仅框架，`init()` 为空，无实际实现  
**实现策略**:
- 采用 `@hono/node-ws` 官方库，协议兼容 OpenAI SSE 格式
- 心跳机制：30s ping，15s 超时断开
- 支持流式取消（abort signal 转发到 Provider）

## 四、关键架构决策记录 (ADR)

### ADR-001: 存储层选型

| 存储场景 | 技术选型 | 说明 |
|---------|---------|------|
| 限流 / 缓存 | **Redis** | 原子操作、过期机制原生支持 |
| 指标 / 用量 | **Redis TimeSeries** | 高写入吞吐，自动聚合 |
| 请求日志 / 审计 | **ClickHouse** (v1.1) | 列式存储，分析查询快 100x |
| 租户 / 配置 | **文件系统 → PostgreSQL** | v1.0 用文件，v1.1 考虑迁 DB |

> ✅ **当前决策**: v1.0 全量用 Redis，按功能分 DB（0: 缓存, 1: 限流, 2: 指标, 3: 健康状态）

---

### ADR-002: 流式协议选择

**选项**:
- A: WebSocket 全双工
- B: SSE 服务器推送事件
- C: 两者都支持

**决策**: ✅ **默认 SSE，可选 WebSocket**

**理由**:
1. SSE 是 OpenAI 标准协议，客户端兼容无需改造
2. SSE 代理友好 (Nginx/Cloudflare 原生支持)
3. WebSocket 作为高级功能供实时协作场景使用
4. 两者共享相同的消息分片逻辑，代码复用率 80%+

---

### ADR-003: 语义缓存实现路径

| 阶段 | 方案 | 复杂度 | 相似度精度 |
|------|------|--------|-----------|
| v1.0 | **Token Jaccard 相似度** | 低 | ~75% |
| v1.1 | **本地轻量 Embedding (all-MiniLM)** | 中 | ~90% |
| v1.2 | **Provider Embedding API** | 高 | ~95% |

**决策**: ✅ v1.0 采用 Token Jaccard，零额外依赖，满足 80% 场景

1. **CI 先行原则**: 所有代码必须在 CI 保护下合并
2. **测试驱动原则**: 新功能必须附带测试，老功能补测试后标记完成
3. **文档同步原则**: 每个功能完成时同步更新 README / API 文档

5. **镀金项延后原则**: 所有非核心功能先做标记，v1.1 版本再考虑
6. **无模拟数据原则**: 前端 Phase 2 开始全部接入真实 API，用 MSW 替代硬编码 mock
